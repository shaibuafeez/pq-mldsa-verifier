# Security

> **Status: experimental, unaudited.** This software has not undergone an
> independent security review. Do not use it to protect real assets without one.

## Reporting a vulnerability

Please report security issues privately to the maintainer (open a GitHub
security advisory on this repository) rather than filing a public issue.
Coordinated disclosure is appreciated.

## What is verified, and how

The cryptography in this repo is checked at several levels:

- **Real-vector verification.** `MLDSAVerifier.verify` accepts a genuine
  ML-DSA-65 signature generated by `@noble/post-quantum` — the same
  `js-noble-vectors.json` that `multivmlabs/pq-smart-wallet`'s Rust Stylus
  verifier is tested against. Wrong message and tampered signature return
  `false`. (`test/MLDSAVerifier.t.sol`)
- **Cross-implementation agreement.** Three independent implementations
  (Noble JS, the Rust `ml-dsa` crate, this Solidity) accept the same vector.
- **SHAKE known-answer tests.** SHAKE-128/256 match NIST KATs.
- **NTT round-trip + range** (incl. fuzzing): `INTT(NTT(p)) == p` and all
  coefficients stay reduced mod Q. (`test/Fuzz.t.sol`)
- **Graceful failure.** `verify` never reverts on malformed input; it returns
  `false`. (fuzzed in `test/Fuzz.t.sol`)
- **Real wallet integration.** The unmodified upstream `PQValidatorModule` is
  driven against this verifier end-to-end with no mocks.
  (`test/integration/PQWalletIntegration.t.sol`)

## Threat model

| Component | In scope | Notes |
|-----------|----------|-------|
| `MLDSAVerifier` (full) | Soundness of FIPS 204 verification: must reject every invalid signature and accept every valid one. | The headline path. Deterministic, fully re-executed on-chain. |
| `MLDSAOptimistic` | Fraud-proof correctness: a committed step with a wrong output must be challengeable; an honest commitment must not be slashable. | **Not yet fully sound — see below.** |
| Off-chain `hints/` | Liveness only. A wrong hint output cannot forge a verification; at worst it makes an honest commitment challengeable (a liveness problem, not a safety one). | Parity with on-chain re-execution is enforced by `test/OptimisticHintParity.t.sol`. |

Out of scope: the security of the ML-DSA scheme itself (assumed per FIPS 204),
side-channel resistance (irrelevant on a public chain), and the trustworthiness
of the public key stored by the consuming wallet.

## Known limitations

### 1. Optimistic mode is a PoC — step *linkage* is not enforced (most important)

`MLDSAOptimistic` re-executes **every** primitive step on-chain (ExpandA,
SHAKE-256, SampleInBall, NTT, InvNTT, pointwise mul/add/sub, scale-by-2^d,
UseHint). A challenger can dispute any single step and the contract recomputes
that exact primitive; a mismatch slashes the prover. This is proven in
`test/MLDSAOptimisticChallenge.t.sol`, and on-chain/off-chain agreement on all
102 polynomial steps is proven in `test/OptimisticHintParity.t.sol`.

**What is still missing for full L1 soundness:** the challenge protocol verifies
each step *in isolation*. It does **not** yet enforce *linkage* —

- that step _N_'s committed output is exactly step _N+1_'s committed input,
- that the first steps' inputs are bound to the real `(publicKey, message,
  signature)`, and
- that the final step's output drives the accept/reject decision.

Without linkage, a malicious prover could commit a Merkle tree of
individually-correct primitive steps that do not actually compose into the
verification of the claimed signature, and finalize it unchallenged. Closing
this — e.g. by committing a structured execution trace (each leaf binds
`opcode || inputCommitment || outputCommitment` with explicit wiring to
predecessors and to the public inputs) and checking the wiring during a
challenge — is the next milestone. **Until then, use `MLDSAVerifier` (full)
for anything real.**

### 2. Unaudited

No independent review has been performed. The pure-Solidity Keccak/SHAKE, NTT,
and FIPS 204 arithmetic are the highest-value audit targets.

### 3. Full verification cost

`MLDSAVerifier.verify` costs ~163M gas — above Ethereum L1's block gas limit. It
is intended for L2s with high gas limits, or off-chain/`eth_call` use, until the
optimistic path is made sound (limitation 1) or an EVM SHAKE precompile exists.

### 4. Parameter set

Only ML-DSA-65 (NIST Level 3) is implemented. ML-DSA-44 and ML-DSA-87 are not.
